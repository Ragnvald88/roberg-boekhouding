# Invoice Builder — Two-Panel Live Preview

**Date**: 2026-03-21
**Status**: Approved

## Problem

The current invoice creation dialog is a restrictive form: forced dropdown for klant, forced werkdagen selection, no preview, no manual line items. The user currently creates invoices manually in Excel because the app gives less freedom than a spreadsheet. The invoice PDF template also doesn't match the user's actual invoice design.

## Design

### Two-Panel Invoice Builder Dialog

A large dialog (90% viewport width, 85% height) with two panels:

**Left panel (440px) — Editor:**

1. **Factuurnummer**: text input, auto-populated with next sequential number (`get_next_factuurnummer`), but fully editable/overridable
2. **Factuurdatum**: date picker via shared `date_input()`, defaults to today, updates to last werkdag date when werkdagen are imported
3. **Klantgegevens** section:
   - **Bedrijf / Praktijk**: `ui.input` with autocomplete suggestions from existing klanten (`autocomplete` parameter). Fully free-text — type anything
   - When an existing klant is selected (matched by name), auto-fill all address fields from the klant's stored data
   - **Contactpersoon**: text input
   - **Adres**: text input
   - **Postcode**: text input
   - **Plaats**: text input
   - All fields editable regardless of whether auto-filled or manually typed
4. **Factuurregels** section:
   - Editable table with columns: Datum, Omschrijving, Aantal, Tarief, Bedrag (computed)
   - Each row has a × delete button
   - **"+ Vrije regel"** button: adds an empty row for manual entry
   - **"+ Werkdagen importeren"** button: opens a sub-dialog listing ongefactureerde werkdagen for the matched klant (if any). Selected werkdagen are added as line items with their actual data (datum, activiteit as omschrijving, uren as aantal, tarief). Reiskosten are added as separate rows.
5. **QR-code uploaden**: drag/drop or click area to upload a QR code image from the user's bank app. Any image size accepted, auto-resized to 2.5×2.5cm on the invoice. Stored in `data/qr/` for reuse across invoices.
6. **Totaal**: computed sum of all line items, displayed prominently
7. **Buttons**: "Annuleren" (flat) + "Genereer factuur" (primary teal)

**Right panel (remaining width) — Live Preview:**

- Renders the actual invoice HTML template with current form data
- Scaled with CSS `transform: scale(0.82)` to fit the panel as a mini A4 page
- Paper shadow effect (`box-shadow`) for document appearance
- Updates live as the user types/changes any field (debounced ~300ms via NiceGUI's reactive binding)
- Shows: business info (from bedrijfsgegevens), typed klant info, all line items, computed totaal, BETAALINFORMATIE with QR code
- BETAALINFORMATIE section pinned to bottom of A4 page via `margin-top: auto` in a flex column

### Klant Autocomplete Behavior

- `ui.input` with `autocomplete` parameter listing all klant names
- When user types and the input matches an existing klant name exactly:
  - Auto-fill Contactpersoon, Adres, Postcode, Plaats from klant's stored address (parsed from `klant.adres` field)
  - Load that klant's ongefactureerde werkdagen for the "Werkdagen importeren" button
- When user types a new name (no match):
  - All address fields remain editable, no werkdagen loaded
  - The invoice is created with a reference to the first active klant in the DB (for the FK constraint), but the PDF shows the typed name

### Werkdagen Import Sub-Dialog

When clicking "+ Werkdagen importeren":
- If no klant is matched, show notification "Selecteer eerst een bestaande klant"
- If klant is matched, show a compact dialog listing ongefactureerde werkdagen for that klant:
  - Each werkdag shown as: `datum — uren u × tarief + km km = bedrag`
  - All checked by default
  - "Toevoegen" button adds selected werkdagen as line items to the main editor
  - Each werkdag becomes one "Waarneming" row + one "Reiskosten" row (if km > 0)

### Line Item Data Structure

Each line item in the editor:
```python
{
    'datum': '21-03-2026',      # displayed date
    'omschrijving': 'Waarneming dagpraktijk',
    'aantal': 9,                 # hours, km, or custom quantity
    'tarief': 77.50,            # per-unit rate
    'werkdag_id': 123,          # if from werkdagen import, else None
}
```

Bedrag is always computed: `aantal × tarief`.

### QR Code Handling

- Upload area accepts any image format (PNG, JPG, etc.)
- Image stored in `data/qr/betaal_qr.png` (single file, reused across all invoices)
- On the invoice PDF: rendered at exactly 2.5cm × 2.5cm using CSS `width: 2.5cm; height: 2.5cm; object-fit: contain`
- If no QR uploaded: the QR cell in BETAALINFORMATIE shows empty/placeholder
- Upload persists across sessions (stored on disk, not in memory)

### Live Preview Rendering

The preview is generated by:
1. Rendering the Jinja2 invoice template (`templates/factuur.html`) with current form data
2. The rendered HTML string is set as the content of a `ui.html()` element
3. The element is wrapped in a container with paper styling (white bg, shadow, fixed A4 dimensions)
4. Re-rendered on every form change via a debounced update function

The preview function:
```python
def render_preview():
    html = jinja_env.get_template('factuur.html').render(
        bedrijf=bg_dict,
        klant={'naam': klant_naam, 'adres': full_address},
        nummer=nummer_input.value,
        datum=format_datum(datum_input.value),
        vervaldatum=format_datum(vervaldatum),
        regels=line_items_for_template,
        subtotaal_werk=..., subtotaal_km=..., totaal=...,
        qr_path=qr_path_if_exists,
    )
    preview_html.content = html
```

### "Genereer factuur" Flow

1. Validate: klant name not empty, at least one line item
2. Generate PDF via `asyncio.to_thread(generate_invoice, ...)` using WeasyPrint
3. Save factuur record in DB (`add_factuur()`)
4. If any line items came from werkdagen import: link those werkdagen via `link_werkdagen_to_factuur()`
5. Close dialog, refresh table, show success notification
6. The generated PDF is stored in `data/facturen/`

## Invoice Template Redesign

The current `templates/factuur.html` needs to be updated to match the user's actual invoice design. Key changes:

### Colors
- Change `#1a5276` → `#1a3c5e` throughout (darker navy matching user's invoices)

### Header
- "FACTUUR" title: 27pt (was 18pt), extra-bold, letter-spacing 2px
- "Nummer: YYYY-NNN" format (currently shows as table row)
- Factuurdatum and Vervaldatum as simple right-aligned lines (not table)

### Client Section
- Replace blue-bordered box with plain "Factuur aan:" bold label + address lines
- Address structured as: Bedrijf name, Contactpersoon, Straat + nr, Postcode Plaats

### Line Items Table
- Column widths: Datum 15%, Omschrijving auto, Aantal 10%, Tarief 14%, Totaal 14%
- Header: 8pt, padding 6px
- Body: 9pt, padding 5px
- Last column header: "Totaal" (not "Bedrag")

### Totaal Section
- "TOTAAL" label: 13pt bold
- Amount: 16pt extra-bold
- "* Vrijgesteld van BTW" below: 8pt italic grey

### BETAALINFORMATIE (new section, replaces current payment paragraph)
- Full-width HTML table with dark navy header bar
- Label column (33%): grey background `#f5f7fa`, grey text `#666`
- Value column: white background, semi-bold text `#2c3e50`
- QR code cell: `rowspan="5"`, right side, 2.5×2.5cm (`object-fit: contain`)
- Rows: IBAN, Ten name van, Onder vermelding van (=factuurnummer), Te betalen bedrag, Betaaltermijn (14 dagen)
- Row separators: `#eee` borders

### Footer
- Keep fixed footer with company details
- Add page numbering for multi-page invoices

### Sender Block
- Include IBAN in sender contact info (currently only in payment section)

## Database Changes

### Klant address fields

Currently `klanten.adres` is a single TEXT field containing the full address. For the structured address in the invoice builder, we have two options:

**Option A (recommended)**: Parse the existing `adres` field in the UI when auto-filling. The builder's split fields (contactpersoon, straat, postcode, plaats) are UI-only — they get combined back into a single string for the invoice template. No schema change needed.

**Option B**: Add separate columns to klanten. This would require a migration and updating all pages that use klant.adres. Overkill for this feature.

### QR code storage

No DB change needed. Store as file: `data/qr/betaal_qr.png`. The invoice generator reads this file if it exists.
