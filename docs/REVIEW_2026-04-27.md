# Boekhouding — full code review (round 2)
*Date: 2026-04-27 · Reviewers: Claude Opus 4.7 (3 parallel opus subagents) + Codex CLI 0.125.0 (gpt-5.5, fresh sandbox read-only review).*

> Built on top of `codex.md` (2026-04-25) and `docs/PLAN_2026-04-26.md` (lanes 1-4 shipped, lane 5 deferred). Test baseline: **905 passed, 14 skipped**.

The previous review focused on data-integrity/year-lock holes and UI plumbing. This second pass looked deeper into the **fiscale engine** — and Codex turned up four genuine **rekenfouten** that the first pass and the unit-test suite both missed. Those are the headline of this report.

## Rating

| Domain | Grade | One-line verdict |
|---|---|---|
| Data integrity (year-lock, mutations) | **B+** | Lanes 1-4 closed the obvious holes; 4 unguarded mutation paths and 2 partial guards remain |
| Fiscal engine | **C** | Main waterval is correct; KIA above-band, AHK afbouw scope, MKB-bij-verlies, en tariefsaanpassing-scope zijn fout — could change real aanslag totals |
| **Privé/zakelijk scheiding** | **C-** | Marking a bank-tx as "privé" hides it from /kosten but leaves the linked uitgave silently in the aangifte total |
| UI/UX (NiceGUI patterns) | **B** | Mostly clean; werkdag-tarief overwrite, lazy-create-on-open, missing YearLocked notifications are real |
| Tests | **A-** | Suite is mature and regression-driven; gaps are concentrated in 2025-3-bracket boundaries, KIA above ondergrens, partner-toedeling, oude-werkdag-unlink, en herinneringdatum-via-UI |
| Architecture | **B-** | `database.py` 3,785 regels en `pages/facturen.py` 1,909 regels zijn god-modules. Splitting is desirable but not blocking. |
| **Overall** | **B-** | Production-usable for the current user *if* the fiscale rekenfouten get fixed before tax filing. Without those fixes the tool reports plausibly-wrong totals to a self-employed user who is not an accountant. |

The biggest structural risk is not in the code shape — it is the gap between what `database.py` returns to the fiscale engine versus what the fiscale engine assumes. `genegeerd=1` was wired into `/kosten` and the dashboard but never propagated to `get_uitgaven_per_categorie` and `get_representatie_totaal`, which feed `/aangifte`. The same kind of source-of-truth drift caused Lane 2's health-alert SQL gap. As more invariants get added, this gap will keep biting until the queries are unified.

## A. Verified bugs at HEAD (post lane 1-4)

Severity legend: **P0** = silently produces wrong fiscal totals; **P1** = data corruption or year-lock violation; **P2** = UX or code smell.

### A1 · P0 — Aangifte counts genegeerd=1 bank uitgaven as bedrijfskosten *(Codex new)*
- `database.py:1563-1574` (`get_uitgaven_per_categorie`) and `database.py:2603-2618` (`get_representatie_totaal`) sum `uitgaven.bedrag` with no join to `banktransacties.genegeerd`.
- `database.py:1706-1734` (`mark_banktx_genegeerd`) refuses to set `genegeerd=1` only on `koppeling_type='factuur'` rows. A debit row with a linked **uitgave** can still be flipped to genegeerd.
- **Exploit**: user lazy-creates an uitgave from a debit (cat='Telefoon', €50), then later marks the bank-tx as "privé". The bank-tx disappears from `/kosten` (correct), but the uitgave remains and `fetch_fiscal_data` ⟶ `bereken_volledig` includes the €50 as zakelijke kosten (wrong).
- **Reach**: every aangifte for any year where the user used the "privé"-toggle on a bank-tx that already had an uitgave linked.
- **Fix shape**: either (a) tighten `mark_banktx_genegeerd` to also delete (or at least cascade-mark) the linked uitgave, or (b) add `LEFT JOIN banktransacties b ON b.id = u.bank_tx_id WHERE b.genegeerd = 0 OR u.bank_tx_id IS NULL` to both fiscale queries. (b) is safer because it's read-only — no risk of losing a categorisation the user expects to keep.
- **Tests to add**: `test_aangifte_excludes_genegeerd_uitgave`, `test_get_uitgaven_per_categorie_filters_genegeerd_bank`, `test_get_representatie_totaal_filters_genegeerd_bank`.

### A2 · P0 — KIA wordt €0 boven `kia_bovengrens`; plateau + afbouwband ontbreken *(Codex new)*
- `fiscal/berekeningen.py:271-277`:
  ```python
  if d_kia_ondergrens <= d_invest <= d_kia_bovengrens:
      d_kia = d_invest * D(params['kia_pct']) / D('100')
  ```
  Buiten die band: `d_kia = 0`.
- Belastingdienst KIA 2025 heeft drie banden boven de ondergrens: 28% × investering tot **€70.602**, dan een vast plateau van **€19.769** tot **€130.741**, dan een **afbouw** met -7,56% per euro tot **€387.580**, dan 0.
- **Exploit**: een huisarts die in één jaar voor €100k investeert (zeldzaam maar mogelijk: nieuwe auto + complete praktijkverbouwing) krijgt **€0** KIA in de plaats van **€19.769**.
- **Fix shape**: vervang de cliff door een bracket-functie analoog aan de arbeidskorting-brackets. Drie nieuwe `fiscale_params`-velden zijn nodig (`kia_plateau_bedrag`, `kia_afbouw_eind`, `kia_afbouw_pct`) of stop ze in een JSON-bracket.
- **Tests to add**: `test_kia_at_70602_returns_pct`, `test_kia_at_70603_returns_plateau`, `test_kia_at_130742_starts_afbouw`, `test_kia_at_387580_returns_zero`.

### A3 · P0 — AHK afbouw is op verzamelinkomen exclusief Box 3 *(Codex flagged, partially correct)*
- `fiscal/berekeningen.py:336-339` rekent `verzamelinkomen` als Box 1 only (belastbare_winst − aov − lijfrente, plus optioneel ew_saldo). Box 3 wordt later los berekend en bij de IB-eindsom opgeteld.
- Codex citeert dat AHK sinds 2025 op `verzamelinkomen` (incl. Box 3) wordt afgebouwd.
- **Status DEBATED — needs verification before fix.** De Belastingdienst 2025-AHK-tabel beschrijft afbouw op "belastbaar inkomen uit werk en woning" (Box 1) niet expliciet op verzamelinkomen-incl-Box-3. **Action: lookup officiële wettekst + Boekhouder-praktijk before patching.** Als codex klopt, is dit een over-grant van AHK voor users met Box 3 vermogen >drempel.
- **Fix shape (if confirmed)**: pass `verzamelinkomen_inc_box3 = belastbare_winst_box1 + voordeel_box3` to `bereken_algemene_heffingskorting`. Today the page applies AHK at `pages/aangifte.py:843` after Box 3 totals are known — easy patch point.

### A4 · P0 — Tariefsaanpassing dekt alleen ZA + SA + MKB *(Codex new)*
- `fiscal/berekeningen.py:343-372`:
  ```python
  d_deductions = d_za + d_sa + d_mkb
  ```
  Belastingdienst regel "afbouw tarief aftrekposten bij hoog inkomen" (Wet IB art 2.10/2.10a) geldt ook voor:
  - hypotheekrente eigen woning,
  - persoonsgebonden aftrek (alimentatie, charity, scholing),
  - lijfrente.
- **Exploit**: huisarts boven schijf-3-grens (~€76.817 in 2025) met hypotheek krijgt vol tarief-aftrek (37,48 %) waar het 35,82 % had moeten zijn. Verschil: ~166 bp × hypotheekrente. Voor €30.000 rente per jaar = ~€500 te weinig belasting betaald. Belastingdienst corrigeert dat op de aanslag — kost de user de teruggave + eventueel rente.
- **Fix shape**: include eigenwoning-saldo, persoonsgebonden aftrek, en lijfrente in `d_deductions` voor de tariefsaanpassing-berekening. Watch out: if the deduction is a NEGATIVE number (loss-creating, like rente > forfait → ew_saldo < 0), the formula reverses sign — careful unit tests required.

### A5 · P0 — Volledige ZA bij verlies + MKB-bij-verlies ontbreekt *(Codex new)*
- `fiscal/berekeningen.py:283-307`. ZA wordt onvoorwaardelijk genomen (line 287-294). MKB rekent alleen tegen positieve `na_oa` (line 303: `max(0, d_na_oa) * d_mkb_pct`). Belastbare winst gefloord op 0 (line 307).
- Belastingdienst regel: MKB-vrijstelling moet ook tegen verlies (vermindert het verlies, art 3.79a Wet IB). En ZA bij verlies wordt verrekend via verliesverrekening (carry-forward).
- **Effect**: dit raakt alleen first-year of slecht-jaar scenarios. Voor een vaste huisarts met stabiele omzet: zeldzaam. Maar de code geeft de user een "te grote" verliescarry-forward (geen MKB-shaving) en een belastbare winst van 0 in plaats van een correct uitgerekend verlies.
- **Fix shape**: laat `bereken_volledig` ook negatieve `na_oa - mkb` doorrekenen; UI moet wel uitleggen dat MKB-bij-verlies juist `min(0, na_oa) * mkb_pct` is (verlies verminderen).
- **Verdict**: P0 in waarheid, maar P1 in praktijk-kans-voor-deze-gebruiker. Schedule with the other fiscal fixes.

### A6 · P1 — Werkdag-edit overschrijft historische tarief *(Codex 2026-04-25 + Opus #2 confirmed)*
- `components/werkdag_form.py:171` (`tarief_input.value = k.tarief_uur`) en line 218 (`await _load_klant_data(werkdag.klant_id)` in edit-mode). Line 227 herstelt km, niet tarief.
- **Exploit**: klant tarief was €110 in 2024, gebruiker veranderde 't naar €120 in 2026. Open een 2024-werkdag in edit, save zonder verder iets te doen → werkdag.tarief flipt van 110 naar 120 in DB, omzet voor 2024 verschuift. Als 2024 nog niet definitief is: stille mutatie.
- **Fix shape**: na `_load_klant_data` in edit-mode: `tarief_input.value = werkdag.tarief`. 5 regels code + één regression test.

### A7 · P1 — `set_afschrijving_override` + `delete_afschrijving_override` zonder year-lock *(Codex 2026-04-25 + Codex 2026-04-27 + Opus #1+#2 confirmed)*

- `pages/kosten_investeringen.py:180-205`: UI-lock check is `y < huidige_jaar`, niet `jaarafsluiting_status='definitief'`.
- **Exploit**: huidige jaar markeren definitief, vervolgens een afschrijving-override invoeren — slaagt zonder fout. Verandert fiscale uitkomst van een year die formeel afgesloten is.
- **Fix shape**: 2 regels in DB (`await assert_year_writable(db_path, jaar)` boven elke functie) + UI lock-check leest `jaarafsluiting_status`.

### A8 · P1 — `add_aangifte_document` + `delete_aangifte_document` zonder year-lock *(Opus #3 new)*

### A9 · P1 — Concept-delete en `save_factuur_atomic` step 1 unlinken locked-year werkdagen *(Codex 2026-04-27 + Opus #2 confirmed)*

### A10 · P1 — `apply_factuur_matches` vertrouwt stale proposal datum *(Codex 2026-04-27 new)*
- `database.py:3170, 3196`: year-lock check gebruikt `proposal.bank_datum` ipv re-read van `banktransacties.datum`.
- **Exploit**: vooraf gegenereerde proposals worden later aangewend — als een match-proposal stale bank_datum heeft (test-scenario, hand-gemaakt, of uitgesteld bevestigen), kan een locked-year bank-tx alsnog gemuteerd worden.

### A11 · P1 — Herinnering versturen schrijft `herinnering_datum` zonder year-lock *(Codex 2026-04-27 new)*

- **Fix shape**: routeer via een nieuwe `update_factuur_herinnering_datum(db_path, factuur_id, datum)` die `assert_year_writable` doet op de factuur-datum. (Of accepteer dat herinnering-datum metadata is en de fiscale telling niet raakt — gerechtvaardigd; bespreek met user.)

### A12 · P1 — `delete_klant_locatie` muteert historische werkdagen.locatie_id *(Codex 2026-04-27 new)*

- **Impact**: zacht — locatie_id muteren raakt geen fiscale velden direct (km/uren/tarief blijven), maar de werkdag rapporteert na delete geen locatie meer in oude PDFs/snapshots.

### A13 · P1 — Aangifte/Instellingen save handlers vangen `YearLockedError` niet af *(Codex 2026-04-25 + Opus #1+#2 confirmed)*

## E. Single most important takeaway

**Lane 6 is the slice that actually matters.** Lanes 1-4 closed obvious data-correctness holes; lanes 7-9 close the remaining year-lock and UI seams. But Lane 6 is the only slice that changes what the app *says about your taxes*. If you only do one lane this round: ship Lane 6 first, with the new tests, and then reverify against your Boekhouder report for at least one closed year (2024) before trusting the engine in 2026's aangifte.

Practically: A1 (privé-uitgaven contamineren aangifte) is the highest-blast-radius item — it produces visibly wrong totals every aangifte you've already done if any bank-tx + linked uitgave has been marked privé. A2 (KIA cliff) only bites if you cross €70.602 in one year's investeringen. A4 (tariefsaanpassing scope) bites every aangifte you do in schijf 3 with a mortgage. A5 (MKB-bij-verlies) only bites in loss years.

A1 + A4 together likely change the outcome on actual aanslagen for the user; they should be patched first.
