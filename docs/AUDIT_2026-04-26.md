# Audit Report — Boekhouding App
*Date: 2026-04-26 · branch: kosten-categorisation-consolidation*

## Scope & method
- Three parallel codebase explorations (dashboard/UX, fiscal/year-lock/PDF bugs, code-quality + missing features)
- Codex CLI broad review attempted but timed out without flushing output; the three Explore agents covered the same surface
- Web research on NL ZZP accounting tools (Moneybird/Jortt/e-Boekhouden) and Peppol/UBL e-invoicing for differentiator gaps
- Test suite baseline: **884 passed, 14 skipped** (clean)
- Codebase: ~17,950 LOC Python (excl. .venv, .git, worktree), 11 routes, no orphans

Findings are filtered for *significant* impact — purely aesthetic nits and speculative refactors are intentionally omitted.

## A. Bugs & inconsistencies (verify, then fix)

### A1. Concept facturen leak into `get_data_counts` *(P1 — wrong number)*
`database.py:2538` — counts `n_facturen` without `status != 'concept'` filter. Every other revenue/KPI query correctly excludes concept; this one doesn't, so dashboard "data counts" inflates by drafts. Single-line fix.

### A2. No staleness escalation on long-standing concept invoices *(P1 — UX miss)*
`database.py:2642` — concept facturen always render as `info` alert. A concept open >14 days is forgotten work, not a status-bar info item. Should escalate to `warning` after a threshold.

### A3. No "stale werkdagen" health alert *(P1 — missed revenue)*
`database.py:2591–2669 (get_health_alerts)` — checks uncategorized bank, overdue invoices, concept invoices, missing fiscale_params. **Doesn't** flag werkdagen >30 days old without `factuurnummer`. For a single-user practice this is the single most valuable alert: "you have 5 werkdagen from februari that aren't invoiced yet." Currently those days are visible only on `/werkdagen` if you happen to scroll.

### A4. No "stale verstuurd factuur" reminder trigger *(P1 — cash-flow visibility)*
`get_health_alerts` flags overdue at the 14-day vervaldatum cutoff but doesn't escalate the >30/45/60-day case to suggest "stuur herinnering." The mail compose helper exists; the alert side does not.

### A5. Modal close-after-refresh window in invoice builder *(P2 — fragile)*
Confirmed by CLAUDE.md itself (`pages/facturen.py` invoice builder). On `genereer_factuur` and `opslaan_als_concept`, `on_save()` (refresh_table) runs *before* `dlg.close()`. Stale `pdf_pad` can flash in the row-menu during the refresh window, leading to "PDF niet gevonden" clicks on files `save_factuur_atomic` just rewrote. This is documented as an invariant — verify it's actually held in the current code, since one regression here is silent.

### A6. `bedrag` sign not validated in `add_uitgave`/`update_uitgave` *(P2 — invariant fragility)*
`database.py:1416, 1489` — the codebase's sign convention says `uitgaven.bedrag` is stored positive (`Enforces uitgave.bedrag = ABS(bank_tx.bedrag)` at lazy-create, line 1443). But neither `add_uitgave` nor `update_uitgave` validates the incoming sign. A direct CLI/test/script edit can write a negative bedrag and `get_kpi_kosten`/breakdown queries (which use `ABS()`) will silently flip into wrong totals only on the rows that don't use ABS (e.g. `get_uitgaven_per_categorie` at `database.py:1539`, `get_representatie_totaal` at 2557). Either enforce sign at the writer or use ABS uniformly at all readers. Currently it's mixed.

### A7. Year selector resets between pages *(P2 — UX)*
Each page (`dashboard.py:50`, `werkdagen.py:123`, `kosten.py:51`, `facturen.py`, etc.) has an independent `jaar_select`. Pick 2025 on dashboard → navigate to facturen → defaults back to 2026. No persistent year context. Fix in `components/layout.py` via `app.storage.user` or query-param threading.

### A8. Click-throughs from dashboard alerts are unfiltered *(P2 — UX)*
`dashboard.py:573–615` — "5 werkdagen ongefactureerd" → routes to `/werkdagen` with no filter; user sees all rows and re-filters manually. Same for "12 facturen openstaand" → `/facturen` shows all 200 facturen, not the 12 overdue. Threading `?status=...` query-params already exists for `/transacties`; extend to facturen + werkdagen.

### A9. Empty-year onboarding renders three zero charts *(P2 — UX)*
`dashboard.py:249–250, 274–275` — sparklines hidden if all-zero, but the bar/donut/cumulative charts still render with empty axes. New fiscal year (2026 today) shows scary blank dashboard. A minimal "Nog geen gegevens dit jaar — start met /werkdagen of /transacties" placeholder closes this.

### A10. Bedrijfsgegevens not pre-checked before first invoice *(P2 — silent crash)*
First invoice creation crashes silently if `/instellingen` Bedrijfsgegevens are incomplete (CLAUDE.md notes "loud ValueError" but UI surface is missing). Render an error card on dashboard if `bedrijfsgegevens` row is incomplete, with a deep-link to Instellingen.

### A11. `mark_banktx_genegeerd` doesn't surface inconsistency on linked facturen *(P2 — silent state)*
`database.py:1679–1701` — toggling `genegeerd=1` on a banktx that was matched to a factuur changes the tx visibility but the factuur still claims it's matched. User may think a factuur is paid while its bank link is now hidden. Either decline the toggle when linked, or warn-and-cascade.

### A12. Outdated comment in `get_transacties_view` *(P3 — docs only)*
`database.py:3495` references the deleted `get_kosten_view` function. Confuses readers.

### A13. `get_categorie_suggestions` case-folding produces duplicate-case suggestions *(P3 — UX)*
`database.py:1704–1757` — groups on `LOWER(tegenpartij)` but returns `MAX(tegenpartij)`. If history has both "ING Bank" and "ing bank" with different categorisations, the user sees two near-duplicate suggestions in the dropdown.

### A14. No render-time test for `/aangifte` and `/jaarafsluiting` failure paths *(P2 — test gap)*
`tests/` — covers DB-layer thoroughly, but no test for "render `/aangifte` with no fiscale_params row" (CLAUDE.md says it should show an error-card; verify) or "render `/jaarafsluiting` with a partially-filled snapshot" (schema-tolerant render path). One regression here = blank page in production.

### A15. Codex `$parent.$emit` anti-pattern *(P3 — verify)*
`pages/facturen.py:740–790` — `sendmail`, `sendherinnering`, `markverstuurd`, `markbetaald` all use `$parent.$emit` from `q-item`s inside a teleported menu. CLAUDE.md flags this as broken specifically for `q-btn-dropdown`. Whether it works for the `q-btn` + `q-menu` pattern used here is empirical. If `Verstuur via e-mail` works for the user in practice, this is fine; if any of these row-menu actions feel "dead," this is the cause.

## B. Missing features (objectively better, ordered by value)

### B1. Persistent global year context *(HIGH)*
Single `app.storage.user['jaar']` source-of-truth in `components/layout.py`; pages subscribe instead of declaring their own selectors. Fixes A7 + A8 + 5 derivative aesthetic issues.

### B2. "Te verwerken / Openstaand" hero KPI strip on dashboard *(HIGH)*
Today's hero strip = Omzet / Bedrijfswinst / Belasting. The two numbers a working huisarts actually opens the app for — **te factureren werkdagen** (count + estimated value) and **openstaand bedrag** — are buried below the fold in AANDACHTSPUNTEN. Restructure to: `Omzet YTD | Openstaand | Belasting reserveren | Aangifte in N dagen`.

### B3. Tax-set-aside widget *(HIGH)*
A live IB+ZVW estimate based on fiscale_winst-to-date — "reserveer €X dit jaar tot nu toe." `fiscal/berekeningen.py` already produces the year-end calc; expose a YTD view. Saves the "year-end surprise" failure mode that's the #1 reason ZZP'ers move to Jortt/Moneybird.

### B4. Aangifte deadline countdown *(HIGH)*
Single banner/card: "Aangifte 2025 over 31 dagen" until the May 1 (or extended) deadline. Trivial; high signal.

### B5. Herinnering workflow with state *(HIGH)*
The mail-helper for sending herinneringen exists. There's no record of which facturen had a 1e/2e/laatste herinnering sent, or when. Add a `herinnering_log` (factuur_id, type, datum) table and surface "X facturen wachten op herinnering" as a health alert.

### B6. Backup/restore UI *(HIGH)*
`VACUUM INTO` is the documented backup primitive. Currently no user-facing trigger. Add a "Maak backup nu" button in `/instellingen` that writes a timestamped `.sqlite3` to a configurable path (default: SynologyDrive financieel archief). Restore can be a stretch — backup alone is the 80% win.

### B7. Per-customer statement / AR-aging view *(MEDIUM)*
A `/klanten/<id>` detail page showing: total facturen, openstaand saldo, average payment days, last payment date. For a huisarts with 5–15 recurring praktijken, this turns "who haven't I been paid by" into one click. Currently `/klanten` is a flat CRUD list (138 LOC).

### B8. UBL / Peppol e-invoice export *(MEDIUM, deadline-driven)*
NL government policy targets mandatory e-invoicing for B2B by 2030, with structured UBL/XML via Peppol as the standard. Bookkeepers and larger praktijken increasingly request UBL alongside PDF today. Implementation: a second Jinja template `templates/factuur.xml.j2` rendering the existing `factuur` data into Peppol BIS Billing 3.0 UBL; surface via a "Download UBL" row-menu item. No network sending required initially — file export covers 90% of the case.

### B9. Voorlopige-aanslag termijnen surfacing *(MEDIUM)*
`dashboard.py:356–372` already conditionally shows VA termijnen — but only if `va_data['has_bank_data']`. Decouple from bank presence; if user enters VA data in `/instellingen`, show schedule unconditionally. The Belastingdienst pays out / collects in 11–12 monthly installments; "next betaling 27 mei: €1,234" is high-signal.

### B10. Audit log on factuur status transitions *(LOW–MEDIUM)*
A single `events` table (id, ts, actor, entity_type, entity_id, action, detail_json) appended-to from each mutator. Single-user app today, but compliance-friendly and trivially useful for "wait, when did I mark factuur X verstuurd?"

### B11. Bulk-importable expense PDF auto-categorisation hint *(LOW)*
`scan_archive` already ingests PDFs. A "voorgestelde categorie" pulled via `get_categorie_suggestions` (already exists) on import would close the loop. No OCR/LLM needed — tegenpartij-name match suffices for >50% of cases.

### Explicitly NOT recommended (over-engineering for this app)
- **Mileage tracker / GPS** — km is already in werkdagen with km_tarief=0 for ANW; adding a route-history module is hobby-project scope creep.
- **Multi-bank CSV adapters** — single user with single Rabobank account; YAGNI is correct in CLAUDE.md.
- **iDEAL / Mollie integration** — payment links can be pasted into the existing `betaallink` field. Direct integration adds OAuth, webhook, secret-management — out of scope.
- **OCR / vision API for receipts** — categorie-suggestions on tegenpartij is 80% of the value at 1% of the complexity.
- **Belastingdienst XML aangifte export** — accountant-facing format, niche; not worth the schema-rot risk for a single-user app.
- **Recurring invoices / subscriptions** — huisartswaarnemer ANW workflow doesn't need it (line items are per-werkdag).
- **User auth, multi-user, audit ACLs** — explicitly excluded by CLAUDE.md YAGNI.

## C. Dashboard restructure (consolidated)

Current order makes the user scroll past empty/zero content to find the working numbers. Proposed restructure:

1. **Top banner** (only if applicable): aangifte deadline countdown · year-locked badge · bedrijfsgegevens-incomplete warning
2. **Hero KPI row** (4 cards): Omzet YTD vs vorig jaar · Openstaand · Belasting reserveren · Uren-vs-urencriterium
3. **Aandachtspunten** (only if any): te factureren werkdagen >30d · concept facturen >14d · openstaand >30d → herinnering · uncategorized bank · ontbrekende fiscale_params · ontbrekende bonnetjes
4. **Charts** (collapsible / lazy-render): omzet per maand, cumulatief vs vorig jaar, kosten donut. Hide individual charts when their dataset is all-zero.

Implementation: ~150–200 LOC change in `pages/dashboard.py`, no DB changes.

## D. Performance (low priority for single-user)

- `scan_archive()` recursively scans SynologyDrive on every `/transacties` page open. Wrapped in `asyncio.to_thread` so the event loop survives, but holds one thread for seconds. Cache mtime + diff incrementally.
- `get_transacties_view` returns full year unfiltered to the table. Fine for now (5K rows tops); revisit at 20K+.

These are not real problems today and should not be touched until a concrete slowness is observed.
